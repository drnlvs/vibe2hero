m01 — introdução ao node.js

Antes de escrever qualquer linha de código útil, você precisa entender o que o Node.js é — e o que ele não é. Essa clareza vai economizar horas de confusão mais pra frente.

// pareto — foco aqui
Os 20% que importam neste módulo: entender a diferença entre Node vs Browser, saber rodar um script e entender ESM vs CommonJS. O resto é contexto.

01 · teoria — o que é node.js

// teoria objetiva · ~5 min

Node.js é um runtime de JavaScript que roda fora do browser. Simples assim. Antes do Node (criado em 2009 por Ryan Dahl), JavaScript só existia dentro de navegadores. O Node pegou o motor V8 do Chrome — o que transforma JS em código de máquina ultra-rápido — e colocou ele no servidor.

A decisão de design mais importante do Node foi ser non-blocking e event-driven. Enquanto um servidor tradicional (PHP, Java) bloqueia uma thread esperando o banco de dados responder, o Node registra um callback e continua processando outras coisas. Resultado: com uma única thread, ele aguenta milhares de conexões simultâneas.

Isso explica por que Netflix, PayPal e Uber usam Node.js para serviços de alto tráfego. Não é magia — é o modelo correto para aplicações I/O-bound (APIs, bancos, arquivos).

─ · node.js vs browser

Essa comparação é o que mais confunde iniciantes. Ambos rodam JavaScript, mas são ambientes completamente diferentes:

browser
  • Tem window, document, DOM
  • Não acessa o sistema de arquivos
  • Segurança: sandbox restrito
  • Objetivo: renderizar UI
node.js
  • Tem global, process, __dirname
  • Acesso total ao sistema de arquivos
  • Sem sandbox — acesso ao SO
  • Objetivo: servidor, scripts, APIs

─ · commonjs vs esm — qual usar em 2026

Node.js tem dois sistemas de módulos. Isso é uma fonte histórica de confusão:

  • CommonJS (CJS) — o sistema legado. Usa require() e module.exports. Ainda encontrado em muitos projetos antigos.
  • ESM (ES Modules) — o padrão moderno. Usa import e export. Suportado nativamente desde Node 12+.
// regra prática para 2026
Novos projetos: use sempre "type": "module" no package.json e sintaxe ESM. Encontrou um projeto com require()? É CJS legado — não misture.

02 · exemplo real

// exemplo · ~10 min · rode no terminal
Vamos rodar nosso primeiro script Node.js e explorar o ambiente — sem instalar nada além do próprio Node.

Primeiro, instale o Node.js LTS em nodejs.org e verifique:

terminal bash 
# Verificar versão instalada
node --version   # deve mostrar v22.x ou superior (LTS 2026)
npm --version    # deve mostrar 10.x ou superior

Crie um arquivo explorando.mjs e cole o código abaixo:

explorando.mjs javascript · ESM 
// .mjs = JavaScript com ES Modules
// Alternativa: arquivo .js com "type":"module" no package.json

// 1. O objeto global do Node (equivalente ao window do browser)
console.log('Versão do Node:', process.version);
console.log('Sistema operacional:', process.platform);
console.log('Diretório atual:', process.cwd());

// 2. import com ESM (equivalente ao require() do CJS)
import { readFileSync } from 'node:fs';
import { join } from 'node:path';

// 3. No ESM, não temos __dirname — usamos import.meta
const __dirname = new URL('.', import.meta.url).pathname;
console.log('__dirname (ESM):', __dirname);

// 4. process.env — variáveis de ambiente
console.log('NODE_ENV:', process.env.NODE_ENV ?? 'não definido');

// 5. Isso NÃO existe no Node (apenas no browser):
// console.log(window)  // ReferenceError!
// console.log(document) // ReferenceError!

console.log('\n✓ Ambiente Node.js explorado com sucesso!');
terminal bash 
node explorando.mjs

# Saída esperada:
# Versão do Node: v22.x.x
# Sistema operacional: linux (ou win32, darwin)
# Diretório atual: /caminho/para/sua/pasta
# __dirname (ESM): /caminho/para/sua/pasta/
# NODE_ENV: não definido
# ✓ Ambiente Node.js explorado com sucesso!

Perceba: process é o window do Node. Ele dá acesso ao ambiente de execução — variáveis de ambiente, versão, sistema operacional, diretório atual. Você vai usar process.env em absolutamente todos os projetos reais.
